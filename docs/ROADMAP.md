# Voica — план на будущее

Заметки по нереализованному. Текущая версия закрывает основной сценарий (диктовка
→ текст с пунктуацией → буфер + история) на macOS. Ниже — что обсуждали отложить.

## Дистрибуция и подпись

- Сейчас: подпись локальным **self-signed сертификатом** «Voica Self-Signed»
  (`scripts/make-cert.sh`, ключ в login keychain). Даёт стабильную идентичность →
  разрешение Accessibility держится между обновлениями. Но сертификат не доверенный
  системой, поэтому на чужом Маке первый запуск — System Settings → Privacy & Security →
  **Open Anyway** (или `xattr -dr com.apple.quarantine /Applications/Voica.app`).
- **Один сертификат на все Маки.** Отпечаток сертификата зашит в подпись; macOS привязывает
  к нему грант Accessibility. Если собирать релизы на другом Маке с *другим* сертификатом —
  у пользователей грант слетит (другой отпечаток). Чтобы этого не было: экспортировать
  `.p12` (Keychain Access → «Voica Self-Signed» → Export) и импортировать на все машины сборки.
- **Бэкап `.p12` обязателен.** Если потерять сертификат (умер Мак) — обновления можно будет
  выпускать только с новым сертификатом, и всем пользователям придётся один раз заново выдать
  Accessibility. Бэкап (напр. в облаке) избавляет от этого.
- Бесшовная раздача (без «Open Anyway») и полноценный авто-апдейт требуют настоящей подписи:
  - macOS: Apple Developer ($99/год) → Developer ID + нотаризация. Бонус — одна стабильная
    идентичность на всех машинах и в CI, привязана к аккаунту, а не к связке ключей.
  - Windows: сертификат (~$100–400/год) либо почти бесплатно через Azure Trusted Signing.

## Авто-обновления

Лёгкий первый шаг: «проверка обновлений» — приложение читает публичный манифест,
сравнивает версию, при наличии новой показывает кнопку «Скачать». Без авто-установки.

**Подвох приватного репозитория:** у приватного репо GitHub API и файлы релизов требуют
авторизации (анонимный запрос → 404). Значит распространяемое приложение не сможет
анонимно ни проверить версию, ни скачать `.dmg`. **Токен в приложение зашивать нельзя**
(его вытащат и получат доступ к приватному коду).

Варианты (код остаётся приватным):
1. **Отдельный публичный репо только для релизов** (`voica-releases`) — рекомендуется.
   Код приватный, артефакты и манифест — в публичном репо. Приложение проверяет его анонимно.
2. **Статический манифест** (`latest.json`) на GitHub Pages / в публичном Gist;
   `.dmg` — в любом публичном месте.
3. Сделать код **публичным** — тогда родные GitHub Releases работают анонимно.
4. **Бэкенд-прокси** хранит токен (нужна инфраструктура; оверкилл без сервера).

Важно: ссылка на `.dmg` тоже должна вести в публичное место — приватные релизы для
авто-апдейта смысла не имеют.

Пример `latest.json`:
```json
{
  "version": "0.3.1",
  "url": "https://github.com/Inhum/voica-releases/releases/download/v0.3.1/Voica-0.3.1.dmg",
  "notes": "Статус ключа в нативном стиле"
}
```
Логика в приложении: GET манифеста → сравнить с `CFBundleShortVersionString` → если новее,
показать уведомление + кнопку «Скачать» (открывает `url`).

Полноценный «сам обновился»: **Sparkle** (macOS, нужна подпись) или **Velopack**
(умеет и Windows, и macOS из одной системы). Хост — GitHub Releases / Pages.

## Windows и кросс-платформенность

macOS-версия на Swift/AppKit не портируется (UI и системный клей платформенные).
Само ядро тонкое: запись → POST в Groq → буфер/история/хоткей/трей/настройки.

- **Нативный Windows (C#/.NET, WinUI 3 / WPF):** трей `NotifyIcon`, хоткей `RegisterHotKey`,
  микрофон `NAudio`/WASAPI, ключ Windows Credential Manager (DPAPI), история SQLite.
  Лучший нативный вид, но отдельная кодовая база.
- **Один кросс-платформенный код на обе ОС** (тогда Swift убираем): Tauri (Rust+web),
  Electron (тяжёлый), Avalonia (C#), Flutter (Dart).

Поддержка паритета:
- Один кросс-платформенный код → паритет автоматический (главный аргумент за этот путь).
- Две нативные базы → вынести «ядро-спеку» (формат запроса Groq, модель, схема истории,
  retention, промпт) в общий документ; релизы в локстеп (общие версии и CHANGELOG,
  единый чек-лист фич/QA); каждую фичу делать дважды.

iOS — отдельный проект (нет глобальных хоткеев; путь — кастомная клавиатура / share-extension).

## Возможные фичи

- ~~Авто-вставка текста в активное поле~~ — сделано в 0.4.0.
- ~~Словарь терминов (Whisper `prompt`)~~ — сделано в 0.6.0.
- ~~LLM-постобработка / глоссарий замен~~ — сделано в 0.7.0 (исправление терминов через
  qwen3-32b, opt-in). Отдельно остаётся идея **свободного LLM-форматирования** текста
  (абзацы, списки, чистка слов-паразитов) — если появится потребность.
- **Settings во вкладки.** Окно настроек (720px) уже занимает почти всю высоту экрана
  MacBook Air 13" — следующий добавленный элемент не влезет. Разбить на вкладки строго по
  Apple HIG. Референс — Tailscale Settings (разобрано по скриншотам пользователя):
  - **Toolbar-style вкладки** сверху: иконка + подпись, активная подсвечена синим
    (NSToolbar preference-стиль / NSTabViewController с toolbar), окно меняет высоту под вкладку.
  - **Раскладка строк:** справа-выровненные подписи категорий («General:», «Window:») слева,
    чекбоксы колонкой, под каждым — серый поясняющий текст с отступом (у нас хинты уже так);
    кнопки действий («Manage…») выровнены вправо на строке.
  - **Статусы** — цветные точки в нативном стиле: ● зелёная «включено/ок», ● серая «выключено»
    (ложится на наш статус ключа и на verify chat-модели ниже).
  - Вкладки Voica: **General** (вывод, обновления) / **Dictation** (режим, клавиша) /
    **Vocabulary** (словарь + AI-фикс) / **Data** (аудио, retention, delete all).
  - У Tailscale About — вкладка настроек с блоком обновлений (Check Now, last check).
    Вариант для нас: влить About+Updates в вкладку About. Пока решение — About остаётся
    отдельным окном (ок по HIG); пересмотреть при реализации.
- **Verify доступности chat-модели при включении «Fix terms with AI».** Не все читают
  документацию — должно быть интуитивно (Apple Way): при включении тумблера — мгновенная
  проверка лёгким запросом к `qwen/qwen3-32b`; зелёная галка «модель доступна» (в нативном
  стиле, как статус ключа и как в Focus-radio), при 403/ошибке — предупреждение с подсказкой
  «разрешите модель в console.groq.com → Settings → Limits». Паттерн UI уже есть в
  SettingsWindow (statusIcon/spinner у кнопки Test).
- Экспорт истории; поиск по истории.
- **Выбор моделей в настройках**: STT-модель (Whisper-варианты), язык распознавания, а теперь
  и chat-модель для ИИ-исправления (сейчас `qwen/qwen3-32b` захардкожена — у пользователей
  могут быть иные доступные/разрешённые модели в их Groq-org). Требует вкладок в Settings.
- **Кнопка «Сбросить настройки»** — вернуть все параметры к дефолтам, не трогая ключ и историю
  (в отличие от Delete All Data, который стирает всё).
- Авто-разбивка слишком длинных записей на части.
- **Транскрипция файла («Transcribe audio file…»).** Пункт меню: выбрать аудио/видео-файл →
  пересжать через AVFoundation (16 кГц моно) → Groq Whisper → показать в окне результата /
  сохранить `.txt`. Другой режим, чем живая диктовка (утилита пакетной обработки, вывод как
  документ, а не вставка). Скоуп: короткие файлы одним запросом; длинные (лимит Groq 25 МБ)
  требуют пересжатия + нарезки на куски — их пока отдаём отдельному CLI-скрипту (`~/transcribe.sh`,
  ffmpeg+curl, уже есть). Переиспользует `GroqClient`/ключ. Риск — размытие фокуса на диктовке.
