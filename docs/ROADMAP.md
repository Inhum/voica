# Voica — план на будущее

Заметки по нереализованному. Текущая версия закрывает основной сценарий (диктовка
→ текст с пунктуацией → буфер + история) на macOS. Ниже — что обсуждали отложить.

## Дистрибуция и подпись

- Сейчас: `.dmg` с **ad-hoc** подписью, без нотаризации. На чужом Маке первый запуск —
  System Settings → Privacy & Security → **Open Anyway** (или `xattr -dr com.apple.quarantine /Applications/Voica.app`).
- Бесшовная раздача и авто-апдейт требуют code signing:
  - macOS: Apple Developer ($99/год) → Developer ID + нотаризация.
  - Windows: сертификат (~$100–400/год) либо почти бесплатно через Azure Trusted Signing.

## Авто-обновления

Лёгкий первый шаг: «проверка обновлений» — приложение читает публичный манифест,
сравнивает версию, при наличии новой показывает кнопку «Скачать». Без авто-установки.

**Подвох приватного репозитория:** у приватного репо GitHub API и файлы релизов требуют
авторизации (анонимный запрос → 404). Значит распространяемое приложение не сможет
анонимно ни проверить версию, ни скачать `.dmg`. **Токен в приложение зашивать нельзя**
(его вытащат и получат доступ к приватному коду).

Варианты (код остаётся приватным):
1. **Отдельный публичный репо только для релизов** (`voica-releases`) — рекомендуется.
   Код приватный, артефакты и манифест — в публичном репо. Приложение проверяет его анонимно.
2. **Статический манифест** (`latest.json`) на GitHub Pages / в публичном Gist;
   `.dmg` — в любом публичном месте.
3. Сделать код **публичным** — тогда родные GitHub Releases работают анонимно.
4. **Бэкенд-прокси** хранит токен (нужна инфраструктура; оверкилл без сервера).

Важно: ссылка на `.dmg` тоже должна вести в публичное место — приватные релизы для
авто-апдейта смысла не имеют.

Пример `latest.json`:
```json
{
  "version": "0.3.1",
  "url": "https://github.com/Inhum/voica-releases/releases/download/v0.3.1/Voica-0.3.1.dmg",
  "notes": "Статус ключа в нативном стиле"
}
```
Логика в приложении: GET манифеста → сравнить с `CFBundleShortVersionString` → если новее,
показать уведомление + кнопку «Скачать» (открывает `url`).

Полноценный «сам обновился»: **Sparkle** (macOS, нужна подпись) или **Velopack**
(умеет и Windows, и macOS из одной системы). Хост — GitHub Releases / Pages.

## Windows и кросс-платформенность

macOS-версия на Swift/AppKit не портируется (UI и системный клей платформенные).
Само ядро тонкое: запись → POST в Groq → буфер/история/хоткей/трей/настройки.

- **Нативный Windows (C#/.NET, WinUI 3 / WPF):** трей `NotifyIcon`, хоткей `RegisterHotKey`,
  микрофон `NAudio`/WASAPI, ключ Windows Credential Manager (DPAPI), история SQLite.
  Лучший нативный вид, но отдельная кодовая база.
- **Один кросс-платформенный код на обе ОС** (тогда Swift убираем): Tauri (Rust+web),
  Electron (тяжёлый), Avalonia (C#), Flutter (Dart).

Поддержка паритета:
- Один кросс-платформенный код → паритет автоматический (главный аргумент за этот путь).
- Две нативные базы → вынести «ядро-спеку» (формат запроса Groq, модель, схема истории,
  retention, промпт) в общий документ; релизы в локстеп (общие версии и CHANGELOG,
  единый чек-лист фич/QA); каждую фичу делать дважды.

iOS — отдельный проект (нет глобальных хоткеев; путь — кастомная клавиатура / share-extension).

## Возможные фичи

- Авто-вставка текста в активное поле (а не только буфер).
- LLM-постобработка / форматирование (Groq LLM после Whisper).
- Словарь терминов (передавать в Whisper `prompt`, чтобы реже путал англицизмы).
- Экспорт истории; поиск по истории.
- Выбор модели / языка распознавания вручную.
- Авто-разбивка слишком длинных записей на части.

## Открытые вопросы (в т.ч. для монетизации)

- Модель ключа: BYO-key (пользователь приносит свой) vs управляемый бэкенд с подпиской.
- Подпись/нотаризация — когда и за чей счёт.
- Какой путь для Windows/iOS (паритет vs нативность).
- Pro-фичи и сегментация (авто-вставка, LLM-формат, синк истории, командные лицензии).
